# vlg-parser — экспорт Telegram → MDX

> **Важно:** очистка **безвозвратно** удаляет все `content/…/news/*.mdx`. Сначала проверьте план:  
> `python3 vlg-parser/scripts/tg_export_to_mdx.py --dry-run --clean-only`

## Поведение по умолчанию

Перед записью новых материалов скрипт **полностью сбрасывает** предыдущий экспорт новостей:

| Что | Действие |
| --- | --- |
| `content/…/news/*.mdx` | все файлы удаляются (или `CONTENT_DIR` → `…/news/`) |
| `public/covers/news/` | каталог **пересоздаётся** пустым — кладите сюда обложки из парсера |
| `public/covers/*.jpg` и др. | удаляются только имена вида **`<цифры>-<slug>.ext`** (старый «плоский» вывод), демо `news-test.jpg` / `review-test.jpg` не трогаются |
| `.next/cache`, `.next/dev/cache` | очищаются (кэш Next.js / Turbopack) |
| `tsconfig.tsbuildinfo` | удаляется при наличии |

Полное удаление `.next` — флаг `--purge-next` (дольше следующая сборка).

## Папка ChatExport

Положите выгрузку канала **внутрь `vlg-parser/`** — удобные имена: `ChatExport` или `ChatExport_2026-03-20` (как выдаёт Telegram). Внутри должен быть **`result.json`** и каталоги с медиа (`photos/`, …).

Скрипт сам находит подходящую папку: все `vlg-parser/ChatExport*`, где есть `result.json`. Если таких несколько — берётся **самая свежая по дате изменения**.

Явный путь:

- флаг `--chat-export /путь/к/папке`, или
- переменная окружения **`CHAT_EXPORT_DIR`**.

Тяжёлый экспорт в git обычно не кладут — в корневом `.gitignore` есть `vlg-parser/ChatExport*/`.

## Запуск

Из корня репозитория:

```bash
# только очистка (без экспорта)
python3 vlg-parser/scripts/tg_export_to_mdx.py --clean-only

# очистка + ваш код в run_export() в scripts/tg_export_to_mdx.py
python3 vlg-parser/scripts/tg_export_to_mdx.py

# другая папка экспорта
python3 vlg-parser/scripts/tg_export_to_mdx.py --chat-export vlg-parser/ChatExport_2026-03-20
```

Опции: `--no-clean` (как раньше, без стирания), `--no-flat-covers` (не чистить плоские `12345-slug.jpg` в `public/covers/`), `--purge-next`, `--dry-run`.

Через npm: `npm run content:clean` (очистка), `npm run content:export` (очистка + `run_export()`).

Корень контента задаётся так же, как у сайта: переменная **`CONTENT_DIR`** в `.env` / окружении (см. `.env.example`).

## Что делает экспорт (`tg_export.py`)

Читается только **`result.json`**, сообщения с `type: "message"`.

### Теги (frontmatter `tags`)

Для каждого материала — **до 10** тегов из текста (хэштеги без `#`, бренды вроде **VXE**/**V3**, **промо**/**ролик**/**промо-ролик**, **слив**/утечки, сенсоры, MCU, BS…, `PAW…`/`PMW…`, чипы `nRF52xxx`, латинские токены из заголовка и **slug**). Если в посте **есть видеофайл** (как в экспорте Telegram), всегда добавляется тег **«видео»** (даже без этого слова в тексте). **Без** выдуманных общих слов («периферия», «железо», …). В теги **не попадают** «новость»/«новости» и `tg-export`. Правила — в `news_tag_rules.py`.

### Правило «новость»

В экспорт попадают **только** посты, у которых **первый значимый фрагмент текста** — молния:

- символы **⚡** или **⚡️** (U+26A1, опционально U+FE0F);
- либо сущность **`custom_emoji`** с текстом **⚡**.

Пустые/пробельные куски в начале пропускаются. Так отсекаются длинные посты, где молния только в середине.

### Альбомы (несколько фото)

Подряд идущие сообщения с **пустым подписью**, **только фото** (не видео), **`id` = предыдущий + 1**, разница **`date_unixtime`** с предыдущим **не больше 2 секунд** — один материал. Первая картинка → **`cover`**, поле **`images`** и блок **`<ArticleGallery>`** содержат **все** фото альбома (включая первое), без отдельного заголовка «Галерея» (горизонтальный скролл).

### Видео

Если у первого сообщения группы **первый медиафайл — видео** (`media_type: video_file`), в MDX добавляется **`<video controls playsInline preload="metadata" …>`**; при копировании файла расширение приводится к **нижнему регистру** (`.mp4`). Обложка для списка — **`thumbnail`**, если есть.

### Пути

- фото: `public/covers/news/<slug>-cover.<ext>`, галерея: `<slug>-gallery-N.<ext>`;
- видео: `public/news-media/<slug>/<имя>.<ext>`.

Текст из Telegram: переносы строк в plain‑фрагментах превращаются в отдельные абзацы Markdown (`\n\n`), чтобы списки и строки с эмодзи не слипались в одну строку; одиночные `*курсив*` в plain без сущности `italic` в JSON выводятся как `<em>`.

**MDX:** в свободном тексте экранируются `&`, `<`, `>` (чтобы фрагменты вроде `<site.com>` не считались JSX-тегами); в однострочных **bold/italic/strikethrough** ядро тоже проходит через HTML-экранирование; сущность **`link`** всегда даёт Markdown-ссылку `[текст](url)`, а не `<url>`.

Теги **`<video>`** и разметка из парсера обрабатываются как обычный **MDX/JSX** (без `rehype-raw`, он конфликтует с компилятором MDX).

Проверка экранирования: `python3 -m unittest tg_export_escape_test` (из каталога `vlg-parser/`).

Проверка на нескольких постах:

```bash
python3 vlg-parser/scripts/tg_export_to_mdx.py --no-clean --limit 5
```

Полный прогон (после очистки по умолчанию):

```bash
npm run content:export
python3 vlg-parser/scripts/tg_export_to_mdx.py --limit 5000
```

Флаги: `--author "Имя"`, `--chat-export путь`.
